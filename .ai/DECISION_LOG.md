| Architecture | ADR-007: Transport Metadata Separation | Dataset metadata and transport metadata are separate architectural concerns and must evolve independently. | Approved |
| Architecture | ADR-008: Pure Validation Layer | Dataset validation is a read-only evaluation process that always returns a deterministic validation report. Exception handling and orchestration remain outside the validator. | Approved |
| Architecture | ADR-009: Stateless Artifact Resolution | ArtifactManager derives lifecycle from observable filesystem state. Validation sentinel files are performance caches written exclusively by the Validator and never represent mutable application state. | Approved |
| Architecture | ADR-010: Documentation Ownership | Dataset manifests are the exclusive source of documentation metadata. The Dataset Registry owns operational metadata only. Manifest loading remains stateless and deterministic through ProjectPaths. | Approved |
| Architecture | ADR-011: Dataset IO Loading Abstraction | DatasetLoader is a stateless, read-only binary IO abstraction. It handles secure access of pre-resolved ArtifactInventoryEntry objects. Parsing, preprocessing, and split resolution logic is intentionally excluded to preserve IO separation of concerns. | Approved |
| Architecture | ADR-012: Syntactic Parsing Layer | DatasetParser is a stateless orchestration service that delegates execution to ParserRegistry. Parsers operate purely on BinaryIO streams, ensuring parsing remains strictly syntactic. ParsedRecord represents unvalidated, unnormalized, and dataset-agnostic data structures. | Approved |
| Architecture | ADR-013: Canonical Data Contract | DatasetNormalizer establishes Arbiter's first stable internal data contract (NormalizedRecord) which explicitly supersedes ParsedRecord for all downstream systems. The normalization layer enforces strict 1-to-1 conversion without semantic mapping. Provenance metadata is entirely infrastructural. | Approved |
| Architecture | ADR-014: Schema Mapping Layer | TaskRecord is Arbiter's canonical task-level data contract. SchemaDefinition is the exclusive owner of declarative source-to-target mappings. TaskMapping is the immutable executable mapping unit combining one SchemaDefinition with one execution strategy. MappingRegistry resolves immutable TaskMapping objects exclusively. Mapper implementations are execution-only and contain no dataset-specific knowledge. DatasetMapper is a stateless orchestration service. Mapping is strictly iterator-to-iterator, deterministic, and constant-memory. The mapping layer performs schema projection only and never preprocessing, feature engineering, tokenization, batching, caching, filesystem access, or parser interaction. | Approved |
| Architecture | ADR-015: Streaming Filtering Layer | DatasetFilter is a stateless orchestration service. Filtering operates exclusively on immutable TaskRecord objects. Field access is completely abstracted through the FieldSelector interface. Predicate definitions are standalone immutable models using composition rather than inheritance. FilterStep is the immutable executable unit binding one predicate definition with one execution strategy. FilterPipeline represents an immutable ordered execution chain. Filtering is strictly iterator-to-iterator, single-pass, deterministic, and constant-memory. Accepted records preserve their original object identity. Filtering performs selection only and never mutates, enriches, preprocesses, batches, caches, buffers, tokenizes, or performs dataset-specific logic. | Approved |
| Architecture | ADR-016: Deterministic Streaming Partitioning | DatasetPartitioner is a stateless orchestration service. PartitionId is the canonical immutable partition identity. PartitionMapping is the immutable executable partition configuration. PartitionDefinition owns declarative partition configuration. Partitioner implementations are execution-only and stateless. Partitioning is strictly iterator-to-iterator. Assignment is deterministic. Object identity is preserved. No buffering, tee, caching, batching, shuffling, balancing, preprocessing, feature engineering, or dataset-specific logic is permitted. The partitioning layer concludes Arbiter's generic ingestion pipeline. | Approved |
| Architecture | ADR-017: Preprocessing Framework | DatasetPreprocessor is a stateless orchestration service. PreprocessedRecord is Arbiter's canonical preprocessing contract. PreprocessingMetadata is the exclusive owner of preprocessing metadata. PreprocessingStep is the immutable executable preprocessing unit. PreprocessingPipeline is an immutable ordered execution chain. Preprocessors are execution-only and stateless. Every preprocessing stage creates a new immutable PreprocessedRecord. The underlying TaskRecord and PartitionId are always preserved. Processing is strictly iterator-to-iterator, deterministic, single-pass, and constant-memory. No buffering, batching, caching, filesystem access, preprocessing algorithms, feature engineering, or dataset-specific logic are permitted in the framework milestone. | Approved |
| Architecture | ADR-018: Shared Field Resolution & Text Transformation | The preprocessing layer introduces text transformation strictly bound to the existing DatasetPreprocessor architecture. FieldSelector and SimpleFieldSelector are promoted to a shared dataset infrastructure module (src/core/datasets/selectors.py) ensuring canonical declarative field resolution across all layers. Preprocessing stages correctly generate new immutable TaskRecord objects while preserving semantic equivalence. Preprocessors operate exclusively via selectors, eliminating raw field-name lookups and hardcoded schema logic. M3.2 successfully delivers the deterministic, configurable WhitespaceNormalizationPreprocessor as the pipeline's first transformation stage. | Approved |
| Architecture | ADR-019: Shared Transformation Utility and Unicode Normalization | M3.3 introduces Unicode Normalization utilizing purely `unicodedata` avoiding heavy NLP frameworks. It defines a shared private utility `_apply_text_transformation` capturing the uniform process: field resolution, string manipulation, and creating the immutable Pydantic wrapper cleanly. Crucially, validation responsibility strictly transitions directly to the static `UnicodeNormalizationDefinition` configuration constraints instead of duplicating logic inside the preprocessor. Memory operations remain mathematically determinable `O(1)`. | Approved |
| Architecture | ADR-020: Control Character Removal and Predicate Isolation | M3.5 adds control character removal natively relying on Python's `unicodedata.category() == "Cc"`. It safely isolates the retention policy logic via a dedicated predicate `_should_keep_character()`, dynamically protecting explicit formatting (like `\n` and `\t`) configured via `ControlCharacterRemovalDefinition`. The transformer functions as a pure generator expression mapping through the predicate without breaking iteration invariants or allocating memory beyond `O(1)` bounds. | Approved |
| Architecture | ADR-021: Composite Preprocessing Profiles | M3.6 introduces immutable preprocessing configurations. Profiles provide reusable wrappers around standard pipelines, completely decoupling static configuration from runtime streaming mechanisms. DatasetPreprocessor remains entirely untouched maintaining precise single-pass iteration behaviors natively executing identical pipelines generated safely by profiles. | Approved |
